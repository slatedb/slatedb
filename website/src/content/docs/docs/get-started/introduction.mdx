SlateDB is an embedded storage engine built as a [log-structured merge-tree](https://en.wikipedia.org/wiki/Log-structured_merge-tree). Unlike traditional LSM-tree storage engines, SlateDB writes all data to object storage.

## Vision

Object storage is an amazing technology. It provides highly-durable, highly-scalable, highly-available storage at a great cost. And recent advancements have made it even more attractive:

* Google Cloud Storage supports multi-region and dual-region buckets for high availability.
* Major object stores support conditional writes that SlateDB can use for fencing.
* Amazon S3 Express One Zone has single-digit millisecond latency.

We believe the future of object storage is multi-region, low-latency buckets that support atomic CAS operations. Inspired by [The Cloud Storage Triad: Latency, Cost, Durability](https://materializedview.io/p/cloud-storage-triad-latency-cost-durability), we set out to build a storage engine for the cloud. SlateDB is that storage engine.

## Features

| Feature | What it means |
|---------|---------------|
| Zero-disk architecture | SlateDB can run as an in-process storage engine with no local state, no control plane, and no replication protocol. |
| Single writer | SlateDB is designed for one writer per database. Partitioning can be built above SlateDB, and writer fencing protects the object-store state. |
| Multiple readers | Readers on different nodes can read the same SlateDB database. |
| Read caching | In-memory and optional on-disk caches reduce latency and object-store API cost. |
| Snapshot isolation | Readers and writers can work against a consistent database view. |
| Transactions | Transactional writes are supported. |
| Object-store persistence | SlateDB writes data to object storage, so durability, scalability, and availability come from the configured object store. |
| Writer fencing | SlateDB detects zombie writer processes and prevents them from writing to the database. |
| Pluggable compaction | You can choose the compaction strategy that fits the workload. |

## Use Cases

SlateDB is a great fit for use cases that are tolerant to 50-100ms write latency, are tolerant to data loss during failure, or are willing to pay for frequent API PUT calls. Such use cases include:

* Stream processing
* Serverless functions
* Durable execution
* Workflow orchestration
* Durable caches
* Data lakes
