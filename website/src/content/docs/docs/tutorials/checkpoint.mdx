## Creating a Checkpoint

A checkpoint is a durable manifest entry that pins one view of a database. Garbage collection will not remove the manifests and SSTs needed by that view while the checkpoint is live.

There are two ways to create one:

- Use [`Db::create_checkpoint`](https://docs.rs/slatedb/latest/slatedb/struct.Db.html#method.create_checkpoint) from an open writer when you need the checkpoint to include recent in-memory state. `CheckpointScope::All` flushes WALs and freezes the current MemTable before creating the checkpoint.
- Use [`Admin::create_detached_checkpoint`](https://docs.rs/slatedb/latest/slatedb/admin/struct.Admin.html#method.create_detached_checkpoint) when you want an admin process to pin the latest manifest that already exists in object storage. This does not flush an active writer first.

`CheckpointOptions::lifetime` controls expiry. If you leave it unset, the checkpoint has no expiry and must be deleted explicitly. `CheckpointOptions::name` can attach a label for listing and filtering checkpoints. Names are not unique.

## Reading or Restoring from a Checkpoint

A checkpoint is not a second copy of the data. It points at a manifest version and keeps the files behind that version alive.

Use a checkpoint when you need:

- A long-lived read-only view through `DbReader`.
- A stable source for a clone.
- An operational pin before a backup, export, or risky maintenance task.

If you want a new writable database at a checkpointed state, create a clone from that checkpoint. The clone starts with the source checkpoint's state, then writes its own manifest, WAL, and future SSTs under its own path. See [Clones](/docs/design/clones) for the full flow.

:::caution
Deleting or expiring a checkpoint does not delete data immediately. It only removes the pin. Garbage collection decides what can be physically removed.
:::
