By default, SlateDB starts a compactor inside each `Db` instance. That compactor includes a coordinator and an embedded worker. If you want compaction on separate compute, disable the embedded compactor in the writer and run compaction from another process.

At minimum, you need:

1. One writer opened with compaction disabled.
2. One standalone compactor coordinator.
3. Either the coordinator's embedded worker, or one or more standalone workers.

All processes must use the same `ObjectStore` and database path (the path is the prefix inside your object store).

## Run a compactor without a Db

You can run the standalone coordinator with the SlateDB CLI:

```bash
slatedb --env-file .env --path <db-path> run-compactor
```

This starts a coordinator with an embedded worker. It runs until interrupted with Ctrl-C, then shuts down gracefully.

If you want separate coordinator and worker processes, disable the embedded worker:

```bash
slatedb --env-file .env --path <db-path> run-compactor --no-embedded-worker
slatedb --env-file .env --path <db-path> run-worker
```

You can run multiple workers against the same database. They claim different jobs through the `.compactions` state file.

## Operational Notes

Only one writer should own a SlateDB database path, but compaction can run out of process. Make sure all writers that touch the path have the embedded compactor disabled, or you may end up with more compaction work than expected.

Every compactor and worker must use compatible configuration with the writer. This matters for compression features, block transforms, compaction filters, object-store credentials, and separate WAL object stores.

If the standalone compactor falls behind, reads may fan out across more L0 SSTs and sorted runs. Watch compaction metrics and the `.compactions` state when tuning worker count and `CompactionWorkerOptions`.
