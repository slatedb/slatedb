SlateDB has two layers of benchmark coverage:

- [slatedb-bencher](https://github.com/slatedb/slatedb/tree/main/slatedb-bencher) runs end-to-end workloads against an object store.
- Criterion microbenchmarks measure narrower Rust code paths inside the `slatedb` crate.

Use `slatedb-bencher` when you care about object-store latency, write durability, cache behavior, transactions, or compaction throughput. Use Criterion when you are changing an internal iterator, encoding path, batch path, or other tight loop.

## End-to-End Benchmarks

The benchmarker has three workload families:

- `db`: mixed `put` and `get` traffic with configurable key generation, value size, concurrency, duration, block cache size, and durable-write behavior.
- `transaction`: concurrent transactional workloads.
- `compaction`: load test data, run a compaction, and clear the test data afterward.

Run the `db` workload from the repository root:

```bash
cargo run -r --package slatedb-bencher -- db --duration 120
```

The benchmarker uses the same object-store environment loading as the admin CLI. For S3-compatible stores, set the AWS variables expected by `object_store`, such as `AWS_BUCKET`, `AWS_REGION`, `AWS_ACCESS_KEY_ID`, and `AWS_SECRET_ACCESS_KEY`. Add `AWS_ENDPOINT` and `AWS_ALLOW_HTTP=true` when testing against local or S3-compatible endpoints.

The repository also includes shell scripts that run repeatable benchmark suites and write plots, raw data, logs, and `github-action-benchmark` JSON under `target/bencher/results`:

```bash
./slatedb-bencher/benchmark-db.sh
./slatedb-bencher/benchmark-transaction.sh
```

Set `SLATEDB_BENCH_CLEAN=true` if you want those scripts to remove benchmark data from object storage after each run.

## Compaction Benchmarks

A typical compaction benchmark flow loads data, runs compaction, then clears the generated data:

```bash
cargo run -r --package slatedb-bencher -- compaction load
cargo run -r --package slatedb-bencher -- compaction run
cargo run -r --package slatedb-bencher -- compaction clear
```

You can also run compaction against specific sorted runs:

```bash
cargo run -r --package slatedb-bencher -- compaction run --compaction-sources="1,2"
```

## Microbenchmarks

Criterion benchmarks live in [`slatedb/benches`](https://github.com/slatedb/slatedb/tree/main/slatedb/benches). Run them with Cargo:

```bash
cargo bench -p slatedb
```

The microbenchmarks use [Criterion](https://bheisler.github.io/criterion.rs/) and, on supported platforms, [pprof-rs](https://github.com/tikv/pprof-rs) for profiles. They are best for comparing a local branch against a baseline on the same machine.

## Nightly Benchmarks

The [nightly workflow](https://github.com/slatedb/slatedb/actions/workflows/nightly.yaml) runs end-to-end benchmarks, transaction benchmarks, microbenchmarks, and profiler generation.

The end-to-end benchmark jobs run on [WarpBuild](https://warpbuild.com) runners backed by Hetzner machines in Frankfurt. SlateDB uses [Tigris](https://www.tigrisdata.com/) object storage with the `auto` region setting for those runs. The workflow publishes results in the GitHub Actions summary and uploads benchmark data through `github-action-benchmark`.

Microbenchmark profiles are uploaded to [pprof.me](https://pprof.me), with links added to the nightly summary. Treat nightly numbers as trend data. For release decisions, reproduce the workload on hardware and object-store placement that match your deployment.
