SlateDB gives readers a consistent view at a sequence number. A `DbSnapshot` is an in-process, read-only handle pinned to the database state at the moment `snapshot()` was called. Later writes can continue on the main `Db`, but the snapshot keeps reading the older view.

Snapshots support the normal read APIs, including `get`, `get_key_value`, and `scan`. They do not support writes, and they are not durable metadata. If you need a view that survives process restart or pins files for another process, use a [checkpoint](/docs/design/checkpoints/) instead.

## Creating a snapshot

The call is cheap. SlateDB records the current sequence number and returns a `DbSnapshot` handle. While that handle is alive, reads through it use the pinned sequence number.

## Snapshots vs. checkpoints

Snapshots and checkpoints solve different problems:

- A snapshot is local to one running process. It is cheap and short-lived.
- A checkpoint is stored in the manifest. It can be used by `DbReader`, clones, backups, and admin workflows.

Holding many old snapshots can force SlateDB to preserve older versions longer than it otherwise would. Drop snapshot handles when the read is done.
